# 技术约束

本文档记录跨模块的基础技术约束。新增编辑器能力时，优先遵守这里的术语和边界，避免 UI、CLI、Mermaid 源码和内部模型互相污染。

## 应用运行边界

- 前端统一使用 Vite + React SPA。普通网页版只承诺 Mermaid/Markdown/Canvas 项目文档编辑、查看、导入和导出；本地终端和 Pi Agent 属于 Electron 桌面版能力。
- 桌面版只使用 Electron。Electron 负责窗口、真实文件读写、应用状态、本地 PTY 终端、内嵌浏览器、安装包和每窗口 Pi Worker。
- 不再依赖 loopback HTTP、发现文件或轮询命令队列。Pi 使用官方 JSONL RPC，控制面和编辑器文档请求走 Node IPC，再由 preload 暴露最小 API。
- `editor-runtime.ts` 是 Web/desktop 平台能力的唯一前端适配层。组件不能直接调用 Electron、PTY、Pi Worker 或本地持久化 API。

## Pi Agent 能力边界

- Agent 与终端是并列的独立功能。终端继续使用 xterm + node-pty；Agent 不接管、替换或嵌入终端会话。
- 每个 Electron 编辑器窗口拥有独立 Pi Worker；同一个 Pi session 文件不能同时被两个窗口写入。
- 打开的编辑器文档以 React 状态为事实源。Pi 必须先读取 `revision`，再通过语义文档工具提交 replacement、UTF-16 edits、Mermaid ops 或 Canvas ops。
- 一次文档工具调用对应一次 undo 事务；默认保存到当前文件。保存失败时保留已应用的编辑状态和 dirty 状态。
- Scratch 会话不加载项目资源和内置文件/终端工具；进入项目后通过 Pi session fork 迁移对话。
- 项目 `.pi` 资源必须经过 Pi project trust；包安装、敏感路径、项目外写入和高风险 shell 命令必须显式确认。
- Agent 界面只使用 `src/components/ui/` 中的 shadcn 组件组合；图标继续使用 Iconoir。

## 终端能力边界

- 终端是项目级工具面板，不是项目文档类型，也不写入 Mermaid、Markdown 或 Canvas 文件。
- Web 构建不提供真实终端；Web runtime 只能返回“桌面版可用”的 unsupported 状态。
- 桌面终端默认通过 Electron `node-pty` 创建本地 shell，React 侧只通过 `editor-runtime.ts` 发送 open/write/resize/close 请求。
- 终端 shell 只能来自后端受控列表。前端只传 `shellId`，不能传任意可执行文件路径或命令字符串。
- Windows 受控候选包括默认 shell、PowerShell、PowerShell 7、CMD 和 WSL；后端只返回当前系统实际可用的选项。
- 终端默认 cwd 优先使用当前项目根目录，其次使用当前文件所在目录；后端在缺失或无效时回退到用户主目录。
- 终端输入输出不接入 Pi Agent 上下文；Pi 自己的 `bash` tool 与可视终端保持独立。

## 文档命名边界

- 除 `README.md` 这个平台约定入口外，`docs/` 下文档文件名使用中文。
- 文档正文使用中文；代码标识符、命令、环境变量、文件路径、Mermaid 语法关键字和 API 名称按原文保留。
- 新增架构图优先使用 `.mmd`，并在 README 的“架构文档”小节登记。
- 不把 Windows `Zone.Identifier`、构建产物、截图临时文件或本地验收日志放入项目文档目录。

## 项目文档类型边界

- `.mmd` 和 `.mermaid` 是 Mermaid 文档，进入 Mermaid 解析、渲染、画布、布局注释、图片节点和 Pi 实时文档工具。
- `.md` 和 `.markdown` 是 Markdown 文档，进入 Milkdown/Crepe Markdown 视图和源码视图，不进入 Mermaid 图模型。
- Markdown 层级折叠属于项目视图元数据。Electron 统一写入项目根目录 `.mermaid-canvas-editor/markdown-folds.json`，不得写入 Markdown 正文；项目扫描必须隐藏该目录，Web 保持会话内状态。
- `.canvas.json` 是 Canvas 文档，进入独立无限白板画布编辑器，不进入 Mermaid 图模型或源码视图；Pi 通过 Canvas 文档操作修改。
- 左侧项目文件浏览器递归展示上述五类项目文档；图片文件只作为 Mermaid 或 Canvas 画布拖入资源，不作为项目文档列出。
- 离线 `mmm patch` 只支持 Mermaid 文件；应用内 Pi 文档工具可修改 Mermaid、Markdown、Canvas 和浮动 Markdown 文档。

## Mermaid 术语边界

- 内部图模型使用 `node` 和 `edge`。`CanvasNode`、`MermaidGraph.nodes`、布局、选择、几何和连线路由都继续使用图论语义。
- Mermaid 源码层使用 `id`、`label` 和 `shape`。例如 `N1@{ shape: rect, label: "WebUI" }` 中，`N1` 是稳定节点 ID，不是源码关键字。
- AI/CLI 修改 Mermaid 时必须优先用节点 ID 定位对象，不能用 label 作为唯一标识。label 是用户可读文本，可以重复或被频繁改写。
- Web 界面可以向用户展示“节点 ID”，但不应把 Mermaid 源码里的 ID 描述成新的对象类型。
- 新建节点默认 ID 使用 `N1`、`N2`、`N3` 序列。解析已有 Mermaid 时不迁移 `A`、`WebUI`、`Node1` 等历史或用户自定义 ID。

## 视觉主题边界

- 应用级 token 清单以 [主题令牌清单.md](主题令牌清单.md) 为准。新增主题字段、画布视觉字段、终端 ANSI 色、Mermaid 变量映射和对象级覆盖时，必须先归入该 token 体系。
- Web 界面使用暖纸底、近黑细线和少量珊瑚红强调。红色只用于 selected、active、connection target、primary command 和关键反馈，不作为大面积背景。
- 普通 UI 颜色必须优先使用 shadcn token，例如 `bg-background`、`bg-card`、`text-foreground`、`border-border`、`text-muted-foreground`、`bg-primary`。
- 主题设置页只能修改主题 token。业务组件不直接保存用户主题色，也不绕过 `editor-theme.ts` 自行转换 CSS、Konva、Pixi 或 Mermaid 渲染变量。
- 高亮按钮使用 `bg-primary`，图标和文字使用背景色 token，例如 `text-background` 或 `text-primary-foreground`。未激活图标使用 `text-icon`。
- 标准图标按钮统一使用圆形 `size="icon"`，浮动控件不得单独覆盖成矩形。
- 主浮动 UI 的尺寸、网格、热区和位置必须从 `editor-chrome.ts` 派生；业务组件不得各自硬编码 48px/32px 这类 chrome 尺寸。
- 资源管理器、属性、主题、终端、Agent 和浮动 Markdown 必须通过 `WorkspaceFloatingWindow` 与 `WorkspaceWindowHeader` 组合；标题栏拖动、自动隐藏、固定、全屏、关闭、层级和越界恢复由统一外壳负责。短暂菜单使用 `FloatingPopover`，shadcn Dialog、Sheet、Popover 不冒充持久工作区窗口。
- 终端面板外壳必须复用应用 UI token；终端正文只能使用 `terminal` 和 `ansi` token，不固定黑底或硬编码 ANSI 默认色。
- Konva/Pixi 画布颜色必须集中在 `CANVAS_VISUAL_TOKENS` 或同源主题 token，节点、连线、锚点、对齐辅助线、网格和草稿线不能在渲染组件里分散硬编码主题色。
- 手写样式只能作为 shadcn token 的扩展，例如源码面板行背景和渲染视图网格；这些扩展必须继续引用 CSS 变量。
- 图标统一使用 Iconoir。新增按钮、菜单项和工具入口必须沿用现有尺寸、间距和 hover/active 规则。

# 技术选型与库使用

本文档说明项目当前技术栈、关键库的使用边界，以及新增依赖时应遵守的判断标准。具体版本以 `package.json` 为准。

## 总体选型

项目采用“静态 Web 应用 + Electron 桌面能力 + 纯编辑器核心”的分层方式：

- Web 层使用 Vite、React 和 TypeScript，负责界面、视图切换、源码编辑、官方 Mermaid 渲染、Markdown 编辑、终端模拟器外壳和无限画布交互。
- 桌面层默认使用 Electron，负责真实文件读写、项目文件夹扫描、图片资源访问、本地 PTY 终端、窗口控制、内嵌浏览器和每窗口 Pi Worker；Tauri v2 只保留为迁移期回退。
- 编辑器核心逻辑放在 `src/features/mermaid-editor/lib/`，包括解析、序列化、布局、几何、路由、交互状态机、视觉状态和历史记录。
- UI 组件放在 `src/features/mermaid-editor/components/` 和 `src/components/ui/`，不能绕过核心逻辑直接修改图结构、选择、文件或 viewport。

## 核心运行时

| 技术 / 库 | 项目用途 | 使用边界 |
| --- | --- | --- |
| TypeScript | 全项目主要语言 | 业务模型、编辑器命令、运行时适配和测试都应保持显式类型。 |
| React | Web UI 和桌面 WebView UI | 负责状态组合和渲染，不承载图解析、路由、命令事务等纯逻辑。 |
| React DOM | 浏览器挂载入口 | 仅用于 `src/main.tsx` 等应用入口。 |
| Vite | 开发服务和前端生产构建 | 用于静态 SPA 构建；不引入服务端运行时或 API routes。 |
| Electron | 默认桌面壳和原生能力 | 原生文件、窗口、内嵌浏览器、PTY、Pi Worker 放在 Electron 边界；React 侧通过运行时适配调用。 |
| Tauri v2 | 旧桌面壳和迁移期回退 | 只能通过 `tauri:*` 脚本和 Tauri runtime fallback 使用。 |
| `@tauri-apps/api` | 前端调用旧 Tauri 能力 | 只能通过 `editor-runtime.ts` 等适配层间接使用，组件不要直接调用 `invoke`。 |

## 终端

| 技术 / 库 | 项目用途 | 使用边界 |
| --- | --- | --- |
| `@xterm/xterm` | 浏览器/WebView 内的终端模拟器 | 只负责终端显示、输入、光标、选择和 ANSI 渲染，不负责创建 shell 或执行命令。 |
| `@xterm/addon-fit` | 终端尺寸自适应 | 只在终端面板容器尺寸变化时计算 cols/rows，并通过运行时适配通知 PTY resize。 |
| `node-pty` | Electron 侧本地伪终端 | 只在桌面端创建后端受控 shell、读写 PTY 和 resize；Web 构建不包含真实终端能力。 |

## Pi Agent

| 技术 / 库 | 项目用途 | 使用边界 |
| --- | --- | --- |
| `@earendil-works/pi-coding-agent` | Pi 原生 Agent、会话、模型、工具、资源和 RPC | 只运行在 Electron fork 的 Worker 中；React 不直接 import。 |
| Pi JSONL RPC | prompt、流式事件、模型、thinking、session 与 tree 命令 | stdin/stdout 只传官方协议；控制面和实时文档请求使用 Node IPC。 |
| `proper-lockfile` | Pi global/project settings 并发写保护 | 配合 revision 校验和临时文件 rename；不能覆盖其他窗口的新设置。 |
| `typebox` | 编辑器专用 Pi 工具 schema | 只描述 workspace context 与 live document read/apply/reveal 工具。 |
| `react-markdown` + `remark-gfm` | Agent 消息 Markdown 渲染 | 只渲染模型输出；链接通过 runtime 打开，不能导航走应用 WebView。 |

## Mermaid 与画布

| 技术 / 库 | 项目用途 | 使用边界 |
| --- | --- | --- |
| Mermaid | 官方渲染视图和 `themeVariables` 渲染适配 | 用于预览和只渲染图类型；不把 Mermaid 当作完整 AST 编辑器。 |
| React Konva | React 中挂载 Konva 无限画布 | 只承担图形渲染和原始事件桥接；业务判断进入交互状态机。 |
| Konva | Canvas 图形、命中、拖拽和视觉对象 | 节点、连线、锚点、框选、辅助线等视觉结构由几何和视觉层派生。 |
| PixiJS | `.canvas.json` 独立无限白板渲染 | React 只挂载容器和工具栏；Pixi stage 负责形状、文本、图片、连线、viewport 和对象级视觉反馈。 |
| `@dagrejs/dagre` | flowchart 自动布局 | 只用于计算初始/自动布局结果，不作为文档真相来源。 |
| `d3-shape` | 连线路径和曲线辅助 | 只用于几何路径生成，不承载业务连接语义。 |

## Markdown 编辑

| 技术 / 库 | 项目用途 | 使用边界 |
| --- | --- | --- |
| Milkdown | Markdown 编辑器基础能力 | 只用于 Markdown 文档阅读和编辑，不进入 Mermaid 图模型、画布交互或 CLI patch 语义。 |
| `@milkdown/crepe` | Markdown 所见即所得编辑界面 | 用于右侧 Markdown 工作视图，保留表格、代码块、链接等常规能力；AI 能力不启用。 |
| `@milkdown/react` | Milkdown React 生态依赖 | 作为 Milkdown React 包能力预留；组件内优先通过 Crepe 实例挂载，避免把编辑器生命周期散落到业务状态。 |
| `@milkdown/preset-gfm` | GitHub Flavored Markdown 能力 | 由 Milkdown/Crepe 生态使用，用于表格、任务列表等常见 Markdown 语法。 |

## 关键模块使用约束

- `document-kind.ts` 是项目文档类型注册边界。`.mmd` / `.mermaid` 进入 Mermaid 管线，`.md` / `.markdown` 进入 Markdown 管线，`.canvas.json` 进入独立 Canvas/Pixi 管线。
- `markdown-document.ts` 是 Mermaid 画布 Markdown 文档引用的识别、路径归一化、摘要提取、拖拽载荷和固定卡片几何常量来源。摘要不能写回 Mermaid graph。
- `editor-runtime.ts` 是浏览器和桌面平台能力的前端适配层。文件选择、文件读写、图片 URL、项目扫描、终端会话、桌面命令和 Pi Agent 能力都必须经过这里或同层适配模块。
- `terminal-panel.tsx` 只管理 xterm 生命周期、fit、shell 选择、输入输出事件和终端面板 UI；它不能绕过运行时适配直接调用 Electron / Tauri，也不能传任意命令路径。
- `agent-panel.tsx` 是独立于终端的 Pi 工作表面；它只能使用 shadcn UI 组合，并通过 `useAgentSession` 与运行时适配交互。
- `editor-motion.ts` 和 `use-gsap-motion.ts` 是动效系统入口。GSAP 参数必须从主题 motion token 派生，不能在组件里散落写死时长、缓动或距离。
- `canvas-motion.ts` 是无限画布视觉补间的差异判定层。它只能描述新增、移动、删除和高亮等视觉变化，不能提交 graph、selection 或 history。
- `canvas-interaction.ts` 是画布交互状态机。Konva 事件不能直接写 graph、selection、viewport 或 history。
- `canvas-hit-target.ts` 是命中目标转换层。具体 Konva 图形只能提供稳定 hit id，再交给业务层解释。
- `canvas-document-rendering.ts` 是 `.canvas.json` 独立画布的 viewport 裁剪、端点坐标和命中测试来源；Pixi 组件不能各自散落这些计算。
- `canvas-visual-state.ts` 是视觉状态来源。hover、selected、editing、dragging、connection target 等状态不应散落在组件条件分支里。
- `node-geometry.ts`、`subgraph-geometry.ts`、`edge-geometry.ts` 是几何来源。节点 frame、锚点、标签、路由矩形、对齐和命中测试必须从这些模块派生。
- `editor-chrome.ts` 是浮动 UI 尺寸、网格、热区和位置来源。主浮动按钮、侧栏、菜单行和文件树行高不要在业务组件中另起一套尺寸。
- `editor-theme.ts` 是主题编译入口。CSS 变量、Konva/Pixi token 和 Mermaid `themeVariables` 都应由它统一产出。

## 新增依赖原则

- 优先复用已有库和本地模块，避免为小工具函数引入新依赖。
- 涉及 Mermaid 语法、图布局、几何、AI/CLI、文件系统、终端或桌面能力时，新增依赖前必须明确它处在哪个边界层。
- 画布热路径依赖必须评估性能影响，不能在 pointer move、drag、wheel、hover 显隐中引入重计算或异步 I/O。
- UI 依赖必须能融入现有 Tailwind、Radix、Iconoir 和 shadcn 风格组件体系。
- 桌面依赖必须同时考虑 Web 静态构建和 Electron 桌面构建；浏览器路径不能依赖 Node、Electron-only 或 Tauri-only API。
- 画布新建项目文档必须经过运行时适配层。Electron 后端只接受项目根目录内的纯 Markdown 文件名，并使用独占写入拒绝覆盖；Web 和旧 Tauri 返回显式 unsupported。
- 新增依赖后需要补充对应测试、文档说明，以及必要的构建或运行脚本更新。
